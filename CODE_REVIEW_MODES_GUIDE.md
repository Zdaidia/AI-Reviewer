# Code Review 模式使用指南

## 两种模式对比

### 🔵 传统模式 (Traditional Mode)

**工作原理：**
```
规则引擎扫描 → 发现问题 → 在代码中添加 TODO 注释
```

**使用场景：**
- ✅ 快速代码检查
- ✅ 标记代码问题位置
- ✅ 团队协作代码审查
- ✅ 持续集成流程

**实际操作：**
1. 点击"添加 Code Review 注释"按钮
2. 选择"传统模式"
3. 系统会自动在每个问题的代码行添加 TODO 注释

**示例效果：**
```javascript
// TODO: [JS-001] Use '===' instead of '=='
if (name == 'admin') {
  // ...
}

// TODO: [JS-002] Avoid using 'var', use 'const' or 'let'
var count = 0;
```

**结果位置：**
- 在代码文件中可以看到添加的注释
- 在右侧面板的 TODO 标签页可以看到所有问题

---

### 🤖 Agent 模式 (AI 智能分析)

**工作原理：**
```
规则引擎扫描 → 收集项目上下文 → AI 深度分析 → 生成优先级报告
```

**使用场景：**
- ✅ 项目整体质量评估
- ✅ 重构优先级规划
- ✅ 技术债务分析
- ✅ 代码质量改进建议

**实际操作：**
1. 点击"添加 Code Review 注释"按钮
2. 选择"Agent 模式"
3. 系统会进行深度分析并生成报告

**AI 分析内容：**

1. **上下文收集**
   - 扫描项目结构
   - 分析依赖关系
   - 识别架构模式
   - 统计代码指标

2. **问题增强分析**
   ```json
   {
     "ruleId": "JS-001",
     "message": "Use '===' instead of '=='",
     "context": {
       "fileIssues": 5,           // 该文件有 5 个问题
       "fileHasMultipleIssues": true,
       "ruleOccurrences": 12,     // 整个项目出现 12 次
       "isRecurring": true        // 重复出现的问题
     },
     "suggestions": [
       "添加适当的 null 检查",
       "使用可选链操作符 (?.)",
       "提供默认值防止 undefined 错误",
       "此问题在代码中出现 12 次，建议全局修复",  // 上下文建议
       "该文件包含多个问题，建议优先重构此文件"  // 上下文建议
     ]
   }
   ```

3. **智能优先级排序**
   ```json
   {
     "priority": [
       {
         "order": 1,
         "issue": "SEC-001",
         "reason": "重复出现的问题, 安全隐患",
         "score": 180
       },
       {
         "order": 2,
         "issue": "JS-001",
         "reason": "问题集中文件",
         "score": 70
       }
     ]
   }
   ```

4. **总结报告**
   ```
   发现 22 个代码问题，其中 5 个高优先级，12 个中优先级。
   建议优先处理: 重复出现的问题, 问题集中文件
   ```

**结果位置：**
- Agent Chat 窗口会自动打开
- 显示详细的分析报告
- 显示优先级排序
- 显示改进建议

**注意：**
- ⚠️ Agent 模式**不会修改代码文件**
- ⚠️ Agent 模式**不会添加 TODO 注释**
- ✅ Agent 模式只提供分析报告和建议

## 🔄 推荐使用流程

### 方案 1：快速标记（适合日常开发）

1. 使用 **传统模式** 快速标记问题
2. 在 TODO 标签页查看所有问题
3. 逐个修复问题

### 方案 2：深度分析（适合项目重构）

1. 使用 **Agent 模式** 进行全面分析
2. 查看优先级报告，了解技术债务分布
3. 根据优先级制定修复计划
4. 对高优先级问题使用 **传统模式** 添加注释
5. 团队协作按优先级修复

### 方案 3：混合使用（推荐）

**第一步：了解全局**
```
Agent 模式 → 生成分析报告 → 了解项目状况
```

**第二步：优先处理**
```
查看优先级列表 → 识别关键问题 → 制定修复计划
```

**第三步：执行修复**
```
传统模式 → 添加 TODO 注释 → 逐个修复
```

**第四步：验证改进**
```
再次使用 Agent 模式 → 对比改进效果 → 持续优化
```

## 💡 使用示例

### 场景 1：新接手项目

**目标：** 快速了解项目质量状况

**操作：**
1. 扫描整个项目
2. 使用 **Agent 模式** 分析
3. 查看报告：
   - 有多少高危问题？
   - 哪些文件问题最多？
   - 有哪些重复出现的问题？
4. 制定修复计划

### 场景 2：日常代码审查

**目标：** 标记需要修复的问题

**操作：**
1. 扫描当前文件
2. 使用 **传统模式** 添加注释
3. 在 TODO 标签页查看
4. 修复问题后删除注释

### 场景 3：重构前评估

**目标：** 确定重构优先级

**操作：**
1. 使用 **Agent 模式** 分析
2. 查看优先级排序：
   - 哪些文件需要优先重构？
   - 哪些问题最严重？
   - 重构能带来多大改进？
3. 制定重构计划
4. 执行重构
5. 再次分析验证效果

## 🔧 故障排除

### Q: 为什么 Agent 模式没有添加注释？
A: Agent 模式设计上就不会修改代码，只提供分析报告。需要添加注释请使用传统模式。

### Q: Agent 模式运行很慢？
A: Agent 模式需要收集项目上下文和 AI 分析，会比传统模式慢一些。这是正常的。

### Q: Agent 模式报错 "Agent system not initialized"？
A: 检查是否配置了 AI 模型的 API Key（GLM-5、OpenAI 等）

### Q: 两种模式扫描结果不同？
A: 两种模式使用相同的规则引擎，扫描结果应该相同。不同之处在于分析和展示方式。

### Q: 能不能同时使用两种模式？
A: 可以！推荐先用 Agent 模式了解全局，再用传统模式标记问题。

## 📝 总结

**传统模式 = 快速标记工具**
- 适合日常使用
- 快速、轻量
- 直接修改代码

**Agent 模式 = 智能分析助手**
- 适合项目评估
- 深度、全面
- 提供决策支持

**最佳实践：先分析后标记**
1. Agent 模式了解全局
2. 传统模式执行修复
3. 持续迭代改进
