# QA Reviewer 通用组件检查框架

## 设计原则

**核心思想**：不是告诉 AI "分页组件要检查什么"，而是告诉 AI "如何分析任何组件"

## 通用组件检查流程

### 第1步：识别组件类型

通过文件路径和文件名识别组件类型：
```
/lib/components/pagination/pagination.dart → 分页组件
/lib/components/table/table.dart → 表格组件
/lib/components/dropdown/filter_dropdown.dart → 下拉组件
```

### 第2步：分析组件接口

检查组件的输入输出：
- **Props（输入参数）**：组件接收哪些参数？
- **Callbacks（回调函数）**：组件提供哪些回调？
- **Events（事件）**：组件触发哪些事件？

### 第3步：追踪数据流向

使用代码图分析：
1. 组件在哪些地方被使用？
2. 父组件传递了哪些参数？
3. 组件的回调如何处理？

### 第4步：验证功能实现

根据数据流向验证：
- 如果组件有 `onPageChange` 回调 → 检查调用方是否传递页码参数
- 如果组件有 `dataSource` 参数 → 检查数据是否正确绑定
- 如果组件有 `sortField` 参数 → 检查排序字段是否传递

## 通用检查模板

### 模板1：参数驱动型组件（如分页、筛选、排序）

```
检查步骤：
1. 列出组件的所有输入参数（Props/Callbacks）
2. 找到使用该组件的地方（父组件）
3. 检查父组件是否正确传递参数
4. 检查参数流向是否完整（父组件 → 组件 → API）

报告问题格式：
- 如果父组件未传递必需参数 → "父组件XXX未传递YYY参数"
- 如果参数值不正确 → "父组件XXX传递的YYY参数值为ZZZ，可能不正确"
- 不报告"组件缺少YYY功能"
```

### 模板2：数据展示型组件（如表格、列表、卡片）

```
检查步骤：
1. 列出组件的数据源参数（dataSource、items等）
2. 找到数据来源（API调用、状态管理）
3. 检查数据格式是否匹配组件要求
4. 检查空数据、加载状态、错误状态的处理

报告问题格式：
- 如果数据格式不匹配 → "数据格式不匹配，组件期望XXX但实际是YYY"
- 如果缺少空状态处理 → "组件缺少空状态展示"
- 不报告"组件缺少排序功能"（排序通过参数控制）
```

### 模板3：交互型组件（如按钮、对话框、表单）

```
检查步骤：
1. 列出组件的事件/回调（onClick、onSubmit等）
2. 找到事件处理函数
3. 检查事件处理是否完整实现
4. 检查是否缺少必要的验证

报告问题格式：
- 如果点击无响应 → "点击事件未绑定处理函数"
- 如果表单提交缺少验证 → "表单提交时缺少YYY验证"
- 不报告"组件缺少功能"（功能由父组件决定）
```

## 实现方式

### 方式1：在提示词中包含检查流程（推荐）

在 `segment-executor.js` 的提示词中添加：

```
【通用组件审查流程】

当审查通用组件时，请按以下步骤分析：

第1步：分析组件接口
   - 列出组件的所有输入参数（Props、Callbacks、Events）
   - 例如：Pagination(onPageChange, pageSize, currentPage)

第2步：追踪使用关系
   - 找到使用该组件的父组件
   - 检查父组件是否传递了正确的参数

第3步：验证功能实现
   - 如果组件有 onPageChange 回调 → 检查父组件是否调用 API 时传递了页码参数
   - 如果组件有 dataSource 参数 → 检查数据是否从 API 正确获取
   - 不要报告"组件缺少功能"，因为功能通过参数控制

报告格式：
- 父组件未传递参数："父组件XXX未传递YYY参数给组件"
- 参数值不正确："传递的参数值YYY可能不正确"
- 不报告："组件缺少XXX功能"
```

### 方式2：动态生成审查指南（高级）

利用代码图分析组件依赖，动态生成审查重点：

```javascript
// 在 segment-executor.js 中
function generateComponentInspectionPrompt(componentFile, codeGraph) {
  const componentInfo = analyzeComponent(componentFile, codeGraph);

  let prompt = `
【组件分析结果】
组件名称: ${componentInfo.name}
输入参数: ${componentInfo.props.join(', ')}
回调函数: ${componentInfo.callbacks.join(', ')}
使用位置: ${componentInfo.usageLocations.join(', ')}

请根据以上分析，检查功能实现是否正确：
`;

  return prompt;
}
```

## 检查规则速查表

| 组件类型 | 检查重点 | 不检查 |
|----------|----------|--------|
| 分页 | 父组件是否传递页码参数、API调用是否正确传递分页参数 | "缺少分页功能" |
| 表格 | 数据源是否正确、列定义是否匹配 | "缺少排序功能"、"缺少筛选" |
| 筛选 | 筛选条件是否传递、API调用是否正确传递筛选参数 | "缺少筛选功能" |
| 按钮 | 点击事件是否绑定、处理函数是否存在 | "缺少功能" |
| 表单 | 提交事件是否绑定、验证是否完整 | "缺少验证" |

## 示例

### 示例1：分页组件检查

❌ **错误报告**：
```
{
  "message": "缺少分页功能的具体实现",
  "suggestion": "需要实现分页逻辑"
}
```

✅ **正确报告**：
```
{
  "message": "父组件 AccountManagement 在调用 API 时未传递 page 参数",
  "suggestion": "在 fetchAccountList API 调用中添加 page 参数，值从分页组件的 currentPage 获取"
}
```

### 示例2：表格组件检查

❌ **错误报告**：
```
{
  "message": "表格组件未实现排序功能",
  "suggestion": "需要添加排序功能"
}
```

✅ **正确报告**：
```
{
  "message": "表格组件支持排序功能（sortField 参数），但父组件未传递此参数",
  "suggestion": "在 AccountTable 组件中传递 sortField 参数，或在 API 调用时传递排序字段"
}
```

## 后续扩展

添加新组件时，只需：
1. 在 `isGenericComponent` 中添加组件名称关键词
2. 在检查规则速查表中添加一行
3. 在提示词的通用组件审查流程中引用（如需要）

无需修改核心审查逻辑！
